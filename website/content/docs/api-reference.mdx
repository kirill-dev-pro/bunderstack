# API Reference

## createBunderstack(options)

```ts
async function createBunderstack<
  TSchema extends Record<string, unknown>,
  const TAccess extends Record<string, TableAccessInput> | undefined =
    undefined,
  const TStorage extends StorageConfigInput | undefined = undefined,
  const TEnv extends EnvConfigInput | undefined = undefined,
  TRouter extends AnyRouter = AnyRouter,
>(
  options: BunderstackConfig<TSchema, TAccess, TStorage, TEnv> & {
    trpc: (t: BunderstackTRPC<TSchema, ValidatedEnv<TEnv>>) => TRouter
  },
): Promise<
  BunderstackApp<TSchema, TAccess, BucketNamesOf<TStorage>, TEnv, TRouter>
>
```

Also accepts `trpc?: AnyRouter` (prebuilt router) or no `trpc` key at all.
Generics are captured so clients can infer tables, buckets, and the tRPC
router from `typeof app` (see `$inferClient` below).

`createBunderstack` is `async` ‚Äî it loads the database driver as a dynamic
import (optional peer dependencies for SQLite/PGlite/Postgres stay out of
projects that don't need them), so `await` it:

```ts
export const app = await createBunderstack({ schema })
```

### BunderstackConfig

```ts
type BunderstackConfig<TSchema, TAccess, TStorage, TEnv> = {
  schema: TSchema
  access?: TAccess // Record<tableKey, TableAccessInput>
  database?: { url?: string; authToken?: string }
  auth?: BetterAuthConfig // native BetterAuth options (except database)
  storage?: TStorage // StorageConfigInput: backends, defaultBucket, buckets
  email?: EmailConfigInput // { from: string; provider?: 'resend' | 'smtp' | 'console' | adapter }
  env?: TEnv // EnvConfigInput: server + client Zod schemas
  // trpc is declared on createBunderstack's overloads for contextual typing
  rateLimit?: boolean | RateLimitConfig
  idempotency?: boolean | IdempotencyConfig
  realtime?: boolean | { keepaliveMs?: number; bufferSize?: number; redis?: ‚Ä¶ }
}
```

### BunderstackApp

```ts
type BunderstackApp<TSchema, TAccess, TBuckets, TEnv, TRouter> = {
  handler: (req: Request) => Promise<Response>
  // Typed for your schema's dialect: LibSQLDatabase for sqliteTable,
  // PgDatabase for pgTable (see Database dialect below)
  db: DbFor<TSchema>
  auth: Auth
  storage: StorageFacade
  router: Hono
  /** Validated env: base vars + config's `env` extension, fully typed. */
  env: ValidatedEnv<TEnv>
  /** Email facade ‚Äî always present; send() throws if email isn't configured. */
  email: EmailFacade
  /** Raw tRPC router when the config declared one ‚Äî escape hatch. */
  trpcRouter?: AnyRouter
  /** Type-only carrier for client inference ‚Äî never assigned at runtime. */
  readonly $inferClient?: {
    schema: TSchema
    access: TAccess
    buckets: TBuckets
    trpc: TRouter
  }
}
```

Export `type App = typeof app` and the client packages infer everything from
it.

## Database dialect

The dialect is inferred from your schema ‚Äî `sqliteTable` ‚Üí SQLite, `pgTable`
‚Üí Postgres ‚Äî and `DATABASE_URL` picks the engine within it:

| Schema        | `DATABASE_URL`  | Engine                                      |
| ------------- | --------------- | ------------------------------------------- |
| `sqliteTable` | unset           | local SQLite file `./data.db`               |
| `sqliteTable` | `file:./app.db` | local SQLite file                           |
| `sqliteTable` | `libsql://‚Ä¶`    | Turso (`DATABASE_AUTH_TOKEN` for auth)      |
| `pgTable`     | unset           | PGlite in `./data.pglite`                   |
| `pgTable`     | `file:./pgdata` | PGlite in that directory                    |
| `pgTable`     | `memory://`     | PGlite in-memory (tests)                    |
| `pgTable`     | `postgres://‚Ä¶`  | Postgres server (`Bun.sql` / `postgres.js`) |

A URL that contradicts the schema's dialect (e.g. `postgres://` with a
`sqliteTable` schema) throws at `createBunderstack()` time.

`drizzle-orm` is a required peer dependency ‚Äî bunderstack compiles against
your project's own install, so there's no separate copy to keep in sync.
`@libsql/client`, `@electric-sql/pglite`, and `postgres` are optional peers:
only the one your setup needs has to be installed, and a missing one throws
with the exact install command.

**Import Drizzle builders and operators directly from `drizzle-orm`** ‚Äî
`sqliteTable`/`pgTable`/`eq`/`desc`/`sql`/etc. are not re-exported from
`bunderstack`. Two dialect-specific subpaths remain for the pieces that must
be static per dialect: `bunderstack/schema` (sqlite) / `bunderstack/schema/pg`
(Postgres) export the internal file-meta/idempotency tables; `bunderstack/typeid`
(sqlite) / `bunderstack/typeid/pg` (Postgres) export the `typeid()` column
builder.

## bunderstack/provision

```ts
function provision(app: object, options?: { force?: boolean }): Promise<void>
```

Optional entry point ‚Äî import it only if you want Bunderstack to manage the
database on boot. When `<migrations>/meta/_journal.json` exists (folder set by
`database.migrations`, default `./migrations`), applies pending migrations via
drizzle-orm's migrator. Otherwise pushes the schema via drizzle-kit (dev-only
dependency; `bun add -d drizzle-kit`). `force` allows a push that would drop
data. The low-level `provisionSchema(db, schema, options?)` (push only) is
exported from the same entry.

**Postgres + `Bun.sql` + dev push:** drizzle-kit's schema push does not yet
support the `Bun.sql` driver (it assumes raw query results carry a `.rows`
property, which `Bun.sql` doesn't provide) ‚Äî this affects only the no-migrations
dev-push path, not committed migrations. If you hit it, generate and commit
migrations (`bunx drizzle-kit generate`) instead of relying on push, or run
against Postgres via `postgres.js` (Node) during development.

## Client packages

### bunderstack-query

```ts
function createClient<TApp extends AnyBunderstackApp>(options?: {
  baseUrl?: string // default '/api'
  fetch?: (input, init?) => Promise<Response>
  queryClient?: QueryClient // enables mutation cache invalidation
}): BunderstackClient<TApp>
```

Lazy proxy: `api.<table>` (list/get/create/update/delete + TanStack Query
option factories), `api.files.<bucket>` (upload/url/delete), and `api.trpc`
(typed `@trpc/tanstack-react-query` proxy when the server declares a tRPC
router). Also exports `MAX_LIST_LIMIT` (200 ‚Äî the server's per-request list
cap), `InferSelect`, `InferInsert`, and the explicit
`createBunderstackQueryClient` builder.

### bunderstack-sync

```ts
function createSyncClient<TApp extends AnyBunderstackApp>(options: {
  queryClient: QueryClient
  baseUrl?: string
  fetch?: (input, init?) => Promise<Response>
  realtime?: boolean // default: true in the browser, false during SSR
}): BunderstackSyncClient<TApp>
```

Per table: `collection`, `table`, `scopedCollection(options)`,
`collectionByIds(ids)`; plus `files.<bucket>` and `realtime` ‚Äî see
[Sync Collections](/docs/sync-collections).

### bunderstack-start

```ts
bunderstackStart<TApp>(options?): { createQueryClient(); createApi(queryClient) }
createApiHandlers(app): { GET; POST; PATCH; DELETE } // catch-all API route
createIsomorphicFetch(options?): fetch // SSR-aware relative-URL fetch
getSessionUser(app, request): Promise<SessionUser | null>
createStartAuthClient(options?) // BetterAuth browser client
```

## RealtimeFacade

```ts
import { type RealtimeFacade } from 'bunderstack'
```

Exposed as `app.realtime` and `ctx.realtime` (in tRPC, jobs, and cron). Used to publish custom write operations to clients subscribed to realtime tables.

- `publish(table, action, record)`: Takes a Drizzle `table` object (which guarantees the physical table exists in `schema`), an `action` string (like `'create' | 'update' | 'delete'`), and the full `record` (which TypeScript constrains to the table's `InferSelectModel`).

If server realtime is disabled or no broker is available, `realtime.enabled` is `false` and `publish` is a no-op. Publishing is async and fire-and-forget: failure to publish into Redis does not crash the caller.
