# Getting Started

## Install

```bash
bun add bunderstack drizzle-orm @libsql/client
bun add -d drizzle-kit
```

## Write your schema

Import Drizzle builders straight from `drizzle-orm` — bunderstack depends on it as a peer, so your project's own install is the one shared instance. Re-export Bunderstack's internal tables so migrations include file storage metadata:

```ts
// schema.ts
import { sqliteTable, integer, text } from 'drizzle-orm/sqlite-core'

export * from 'bunderstack/schema'

// BetterAuth required tables — copy full definitions from examples/standalone/schema.ts
export const user = sqliteTable('user', {
  id: text('id').primaryKey(),
  name: text('name').notNull(),
  email: text('email').notNull().unique(),
  emailVerified: integer('emailVerified', { mode: 'boolean' })
    .notNull()
    .default(false),
  image: text('image'),
  createdAt: integer('createdAt', { mode: 'timestamp' }).notNull(),
  updatedAt: integer('updatedAt', { mode: 'timestamp' }).notNull(),
})
// ... session, account, verification (same pattern)

// Your tables
export const posts = sqliteTable('posts', {
  id: integer('id').primaryKey({ autoIncrement: true }),
  title: text('title').notNull(),
  body: text('body').notNull().default(''),
  userId: text('userId')
    .notNull()
    .references(() => user.id, { onDelete: 'cascade' }),
  createdAt: integer('createdAt', { mode: 'timestamp' })
    .notNull()
    .$defaultFn(() => new Date()),
})
```

## Create the app

```ts
// bunderstack.ts
import { createBunderstack } from 'bunderstack'
import { provision } from 'bunderstack/provision'
import * as schema from './schema'

export const app = await createBunderstack({
  schema,
  auth: { emailAndPassword: { enabled: true } },
})

/** Type-only handle for client inference — see "Talk to it from the client". */
export type App = typeof app

await provision(app)
```

`provision(app)` keeps the database in sync with your schema. The same line works in development and production — what it does depends on whether a migrations folder exists (see below). Don't want the framework touching your database at all? Skip the import and apply migrations yourself.

## Development vs production

The `migrations/` folder is the mode switch — you never change code or set env vars to move between phases:

| Phase         | State                            | What `provision(app)` does                                               |
| ------------- | -------------------------------- | ------------------------------------------------------------------------ |
| Prototyping   | No `migrations/` folder          | Pushes the schema straight to the database on boot (needs `drizzle-kit`) |
| Stabilized    | `migrations/` committed          | Applies pending migrations on boot — `drizzle-kit` is never loaded       |
| Deploy        | Fresh clone + `migrations/`      | Creates the database file and all tables from the committed migrations   |

**Prototyping.** Right after `bun add -d drizzle-kit`, just edit your schema and restart — the database follows. If a change would drop data, provision refuses and tells you to pass `provision(app, { force: true })`.

**Stabilizing.** When you're done playing with the schema, generate the initial migration:

```bash
bunx drizzle-kit generate
```

From this point on provision stops pushing and only applies the committed files. Every further schema change is an explicit `bunx drizzle-kit generate`, reviewed and committed like any other code.

**Deploying.** Clone, `bun install --production`, start. Applying migrations uses drizzle-orm's built-in migrator, so `drizzle-kit` stays a dev-only dependency — it is never installed on, or shipped to, the server. If you deploy without a migrations folder *and* without drizzle-kit, provision fails with instructions rather than guessing.

Configure `drizzle.config.ts` with `schema: './src/schema.ts'` and `out: './migrations'`. If your migrations live elsewhere, point Bunderstack at them with `database: { migrations: './path' }`.

## Using PostgreSQL

Bunderstack infers the database dialect from your schema: define tables with
`pgTable` instead of `sqliteTable` and the whole stack — CRUD, auth, storage
metadata, provisioning — switches to Postgres. No config key to flip.

```ts
// schema.ts
import { pgTable, serial, text } from 'drizzle-orm/pg-core'

export const posts = pgTable('posts', {
  id: serial('id').primaryKey(),
  title: text('title').notNull(),
})
```

For local development, install PGlite — an embedded Postgres that runs in a
local directory, no Docker or server required:

```bash
bun add -d @electric-sql/pglite
```

With no `DATABASE_URL` set, bunderstack runs PGlite in `./data.pglite`,
creating it on first start — the same zero-setup experience as the SQLite
default above. In production, point the same app at a real Postgres server;
no code changes:

```bash
DATABASE_URL=postgres://user:pass@host/db
```

Under Bun, the built-in `Bun.sql` driver is used automatically (no extra
package). On Node runtimes (e.g. Next.js), install the driver:
`npm install postgres`.

See [Configuration](/docs/configuration) for the full dialect/URL table.

## Serve it

```ts
// server.ts
import { app } from './bunderstack'
Bun.serve({ fetch: app.handler })
```

```bash
bun run server.ts
# GET    /api/posts
# POST   /api/posts
# PATCH  /api/posts/:id
# DELETE /api/posts/:id
# POST   /api/auth/sign-up/email
# POST   /api/auth/sign-in/email
# POST   /api/files
# GET    /health
```

The `posts` table has a `userId` column, so Bunderstack auto-wires it: list/get are public, create sets the owner from session, update/delete require the owner.

## Add access rules

To customize per-operation permissions, use `defineAccess`:

```ts
// access.ts
import { defineAccess } from 'bunderstack/access'
import * as schema from './schema'

export const access = defineAccess(schema, {
  posts: {
    ownerColumn: 'userId',
    list: 'public',
    create: 'authenticated',
    update: 'owner',
    delete: 'owner',
    searchableColumns: ['title', 'body'],
  },
})
```

```ts
// bunderstack.ts
export const app = await createBunderstack({
  schema,
  access,
  auth: { emailAndPassword: { enabled: true } },
})

await provision(app)
```

## Talk to it from the client

Because `App` carries your schema, access rules, and buckets at the type
level, the client needs zero configuration — tables and buckets are inferred:

```ts
// api-client.ts
import { createClient } from 'bunderstack-query'
import type { App } from './bunderstack' // type-only: no server code in the bundle

export const api = createClient<App>({ queryClient })

api.posts.list({ limit: 20 })
api.files.uploads.upload(file)
```

Add a table on the server and the client knows it instantly — no tuples to
keep in sync. See [Query client](/docs/query-client) and
[Sync collections](/docs/sync-collections).
