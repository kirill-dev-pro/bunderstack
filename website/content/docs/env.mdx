# Environment Validation

Bunderstack validates environment variables at boot so your app fails fast with
clear messages instead of failing deep in production with missing values.
Server-side and client-side vars are separate, and `app.env` is fully typed.

## Configuration

Add an `env` key with `server` and `client` sections. Each section is a record
of Zod schemas:

```ts
import { z } from 'zod'
import { createBunderstack } from 'bunderstack'
import * as schema from './schema'

export const app = await createBunderstack({
  schema,
  env: {
    server: {
      STRIPE_API_KEY: z.string(),
      WEBHOOK_SECRET: z.string(),
    },
    client: {
      PUBLIC_APP_URL: z.string().url(),
      PUBLIC_SENTRY_DSN: z.string().optional(),
    },
  },
})
```

### Naming convention

- **Server vars** must NOT start with `PUBLIC_`. Put them in `env.server`.
- **Client vars** MUST start with `PUBLIC_`. Put them in `env.client`.

Bunderstack enforces this at boot and throws a `BunderstackEnvError` listing
all violations.

## Built-in base schema

Bunderstack always validates these vars (no config needed):

| Variable              | Default          | Required in production? |
| --------------------- | ---------------- | ----------------------- |
| `NODE_ENV`            | —                | No                      |
| `DATABASE_URL`        | `file:./data.db` | No                      |
| `DATABASE_AUTH_TOKEN` | —                | No                      |
| `AUTH_SECRET`         | dev default      | **Yes**                 |
| `REDIS_URL`           | —                | No                      |
| `RESEND_API_KEY`      | —                | Only with resend email  |
| `SMTP_URL`            | —                | Only with SMTP email    |

## Using `app.env`

```ts
// Fully typed — server vars AND PUBLIC_ vars are available on the server
app.env.STRIPE_API_KEY // string
app.env.PUBLIC_APP_URL // string
app.env.AUTH_SECRET // string (from base schema)
```

All vars are available in the tRPC context too:

```ts
export const app = await createBunderstack({
  schema,
  env: { server: { API_KEY: z.string() } },
  trpc: (t) =>
    t.router({
      whoami: t.procedure.query(({ ctx }) => {
        return { env: ctx.env.API_KEY } // fully typed
      }),
    }),
})
```

## Client-side (`createClientEnv`)

`createClientEnv` validates only the `client` section, making it safe for
browser bundles. Server keys exist as traps that throw on access:

```ts
// env.client.ts — safe for the browser
import { createClientEnv } from 'bunderstack/env'
import type { env } from './bunderstack' // type-only

export const clientEnv = createClientEnv(envConfig)
// typeof clientEnv === { PUBLIC_APP_URL: string; PUBLIC_SENTRY_DSN?: string }
```

If you use Vite, pass `import.meta.env` as the value source:

```ts
export const envConfig = {
  client: { PUBLIC_APP_URL: z.string().url() },
  runtimeEnv: import.meta.env, // Vite's env object
}
export const clientEnv = createClientEnv(envConfig)
```

## Error type

```ts
import { BunderstackEnvError } from 'bunderstack'

try {
  const app = await createBunderstack({ schema, env })
} catch (err) {
  if (err instanceof BunderstackEnvError) {
    console.log(err.issues) // string[] — one per invalid/missing var
  }
}
```

## Re-using the config type

Keep your env schema next to your Drizzle schema so both server and client can
reference it:

```ts
// env-schema.ts
import { z } from 'zod'
import type { EnvConfigInput } from 'bunderstack'

export const env = {
  server: { STRIPE_KEY: z.string() },
  client: { PUBLIC_APP_URL: z.string().url() },
} satisfies EnvConfigInput
```

```ts
// bunderstack.ts
import { env } from './env-schema'
export const app = await createBunderstack({ schema, env })
```

```ts
// env.client.ts
import { createClientEnv } from 'bunderstack/env'
import { env } from './env-schema'
export const clientEnv = createClientEnv(env)
```
